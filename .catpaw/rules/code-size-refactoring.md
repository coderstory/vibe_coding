# 大文件重构规则

## 触发条件

当新建或修改的代码文件行数超过 **500 行**时，必须执行重构分析。

**适用文件类型：**
- `*.java` - Java 后端代码
- `*.ts` / `*.vue` - TypeScript / Vue 前端代码
- `*.xml` - MyBatis Mapper 文件（超过 300 行时触发）

**不适用：**
- 自动生成的代码
- 第三方库文件
- 配置文件（`.yml`, `.json`, `.properties`）

## 重构分析流程

### Step 1: 评估文件结构

分析文件是否包含以下问题：

| 问题类型 | 症状 | 建议方案 |
|---------|------|---------|
| **职责过多** | 一个类/组件做太多事情 | 按职责拆分为多个类/组件 |
| **方法过长** | 单个方法超过 50 行 | 提取子方法 |
| **重复代码** | 多处相似逻辑 | 提取公共方法/工具类 |
| **深层嵌套** | if/for 嵌套超过 3 层 | 提前返回、策略模式 |
| **参数过多** | 方法参数超过 5 个 | 封装为对象 |

### Step 2: 确定拆分策略

#### Java 后端拆分

```
# Service 类拆分示例
BigService.java (800 行)
    ↓ 拆分为
├── BigService.java (主服务，200 行)
├── BigQueryService.java (查询逻辑，200 行)
├── BigCommandService.java (命令逻辑，200 行)
└── BigValidator.java (校验逻辑，100 行)
```

#### Vue 组件拆分

```
# Vue 组件拆分示例
BigComponent.vue (600 行)
    ↓ 拆分为
├── BigComponent.vue (主组件，150 行)
├── components/
│   ├── BigHeader.vue (头部，100 行)
│   ├── BigList.vue (列表，150 行)
│   └── BigForm.vue (表单，150 行)
└── composables/
    └── useBigLogic.ts (业务逻辑，100 行)
```

### Step 3: 执行拆分

1. **创建新文件** - 按职责划分
2. **迁移代码** - 逐块移动，保持功能完整
3. **更新引用** - 修改 import 语句
4. **运行测试** - 确保功能正常
5. **删除冗余** - 清理原文件中的重复代码

## 评估标准

### 无需拆分的情况

- 文件行数 500-600 行，但结构清晰、职责单一
- 主要是数据定义（如 Entity、DTO、常量类）
- 自动生成的代码（如 Protobuf、OpenAPI 生成的代码）

### 必须拆分的情况

- 文件行数超过 800 行
- 包含多个不相关的业务逻辑
- 存在大量重复代码
- 方法平均超过 30 行

## 拆分目标

| 文件类型 | 目标行数 | 最大行数 |
|---------|---------|---------|
| Java Service | 200-300 行 | 500 行 |
| Java Controller | 100-200 行 | 300 行 |
| Vue 组件 | 150-250 行 | 400 行 |
| TypeScript 模块 | 150-250 行 | 400 行 |
| MyBatis Mapper | 100-200 行 | 300 行 |

## 示例对话

```
用户：帮我修改 WorkflowService，添加新功能

Agent：（检测到 WorkflowService.java 有 850 行）
我注意到 WorkflowService.java 已经有 850 行，建议先进行重构拆分：

当前问题：
- 包含查询、命令、校验等多种职责
- 多个方法超过 50 行
- 存在重复的数据转换逻辑

建议拆分方案：
1. WorkflowQueryService - 查询相关逻辑（约 200 行）
2. WorkflowCommandService - 创建/更新/删除逻辑（约 250 行）
3. WorkflowValidator - 校验逻辑（约 100 行）
4. WorkflowService - 保留核心流程编排（约 200 行）

是否先执行重构？这样可以：
- 提高代码可维护性
- 降低后续开发难度
- 更容易添加新功能
```

## 注意事项

1. **渐进式重构** - 不要一次性拆分太多，每次只处理一个文件
2. **保持兼容** - 拆分后保持原有 API 不变
3. **补充测试** - 拆分后为新模块补充单元测试
4. **更新文档** - 如果是核心模块，更新相关设计文档
