# 代码审查工作流

本技能涵盖代码审查的完整流程：**请求审查** 和 **处理反馈**。

**核心原则：** 早期审查，经常审查。技术严谨性优于社交舒适。

## 一、请求代码审查

### 1.1 何时请求审查

**必需：**
- 子代理驱动开发中的每个任务之后
- 完成主要功能后
- 合并到 main 之前

**可选但有价值：**
- 卡住时（新视角）
- 重构前（基线检查）
- 修复复杂错误后

### 1.2 如何请求审查

**步骤 1：获取 git SHA**
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # 或 origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

**步骤 2：调度 code-reviewer 子代理**

使用 Task 工具和 superpowers:code-reviewer 类型，填写审查模板。

**占位符：**
- `{WHAT_WAS_IMPLEMENTED}` - 你刚刚构建的内容
- `{PLAN_OR_REQUIREMENTS}` - 它应该做什么
- `{BASE_SHA}` - 起始提交
- `{HEAD_SHA}` - 结束提交
- `{DESCRIPTION}` - 简要摘要

**步骤 3：对反馈采取行动**
- 立即修复关键问题
- 在继续前修复重要问题
- 稍后记录次要问题
- 如果审查者错误则反驳（带理由）

### 1.3 请求审查示例

```
[刚刚完成任务 2：添加验证函数]

你：让我在继续前请求代码审查。

BASE_SHA=$(git log --oneline | grep "Task 1" | head -1 | awk '{print $1}')
HEAD_SHA=$(git rev-parse HEAD)

[调度 superpowers:code-reviewer 子代理]
  WHAT_WAS_IMPLEMENTED: 对话索引的验证和修复函数
  PLAN_OR_REQUIREMENTS: docs/plans/deployment-plan.md 中的任务 2
  BASE_SHA: a7981ec
  HEAD_SHA: 3df7661
  DESCRIPTION: 添加了 verifyIndex() 和 repairIndex()，包含 4 种问题类型

[子代理返回]：
  优势：干净的架构，真实的测试
  问题：
    重要：缺少进度指示器
    次要：报告间隔的魔术数字（100）
  评估：准备好继续

你：[修复进度指示器]
[继续到任务 3]
```

### 1.4 与工作流集成

**子代理驱动开发：**
- 每个任务后审查
- 在问题复合前捕获它们
- 在移动到下一个任务前修复

**执行计划：**
- 每个批次（3 个任务）后审查
- 获取反馈，应用，继续

**临时开发：**
- 合并前审查
- 卡住时审查

### 1.5 红旗

**从不：**
- 因为"简单"而跳过审查
- 忽略关键问题
- 继续未修复的重要问题
- 与有效的技术反馈争论

**如果审查者错误：**
- 用技术推理反驳
- 显示证明它工作的代码/测试
- 请求澄清

## 二、接收审查反馈

### 2.1 响应模式

代码审查需要技术评估，而不是情感表演。

```
当收到代码审查反馈时：

1. 阅读：完整阅读反馈而不反应
2. 理解：用自己的话重述要求（或询问）
3. 验证：根据代码库现实检查
4. 评估：对此代码库在技术上是否合理？
5. 响应：技术确认或理性反驳
6. 实施：一次一项，每项测试
```

### 2.2 禁止的响应

**从不：**
- "你完全正确！"（表演性同意）
- "很好的观点！" / "极好的反馈！"（表演性）
- "让我现在实施"（验证前）

**改为：**
- 重述技术要求
- 提出澄清问题
- 如果错误则用技术推理反驳
- 直接开始工作（行动 > 言语）

### 2.3 处理不清楚的反馈

```
如果有任何项目不清楚：
  停止 - 还不要实施任何东西
  在不清楚的项目上请求澄清

原因：项目可能相关。部分理解 = 错误实施。
```

**示例：**
```
你的合作伙伴："修复 1-6"
你理解 1,2,3,6。不清楚 4,5。

❌ 错误：现在实施 1,2,3,6，稍后询问 4,5
✅ 正确："我理解项目 1,2,3,6。需要在继续前澄清 4 和 5。"
```

### 2.4 特定来源处理

**来自你的合作伙伴：**
- **受信任** - 理解后实施
- **如果范围不清楚仍需询问**
- **没有表演性同意**
- **跳到行动** 或技术确认

**来自外部审查者：**
```
实施前：
  1. 检查：对此代码库在技术上是否正确？
  2. 检查：是否会破坏现有功能？
  3. 检查：当前实施的原因？
  4. 检查：在所有平台/版本上是否有效？
  5. 检查：审查者是否理解完整上下文？

如果建议似乎错误：
  用技术推理反驳

如果不能轻松验证：
  如此说明："没有 [X] 我无法验证这个。我应该 [调查/询问/继续] 吗？"

如果与你的合作伙伴之前的决策冲突：
  首先停止并与你的合作伙伴讨论
```

### 2.5 "专业"功能的 YAGNI 检查

```
如果审查者建议"正确实施"：
  在代码库中 grep 实际使用

  如果未使用："此端点未被调用。删除它（YAGNI）？"
  如果使用：然后正确实施
```

### 2.6 实施顺序

```
对于多项目反馈：
  1. 首先澄清任何不清楚的内容
  2. 然后按此顺序实施：
     - 阻塞问题（破坏、安全）
     - 简单修复（拼写、导入）
     - 复杂修复（重构、逻辑）
  3. 分别测试每个修复
  4. 验证无回归
```

### 2.7 何时反驳

当以下情况时反驳：
- 建议破坏现有功能
- 审查者缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对此技术栈在技术上不正确
- 存在遗留/兼容性原因
- 与架构决策冲突

**如何反驳：**
- 使用技术推理，而不是防御性
- 提出具体问题
- 引用工作的测试/代码
- 如果是架构性问题，涉及相关方

### 2.8 确认正确的反馈

当反馈正确时：
```
✅ "已修复。[简要说明改变了什么]"
✅ "好发现 - [具体问题]。已在 [位置] 修复。"
✅ [直接修复并在代码中显示]

❌ "你完全正确！"
❌ "很好的观点！"
❌ "感谢发现那个！"
❌ "感谢 [任何东西]"
❌ 任何感谢表达
```

**为什么不感谢：** 行动胜于言语。只需修复它。代码本身显示你听到了反馈。

### 2.9 优雅地纠正你的反驳

如果你反驳并且错了：
```
✅ "你是对的 - 我检查了 [X] 并且它确实 [Y]。现在正在实施。"
✅ "验证了这个并且你是对的。我最初的理解是错误的，因为 [原因]。修复。"

❌ 长道歉
❌ 为为什么你反驳辩护
❌ 过度解释
```

如实陈述更正并继续。

### 2.10 常见错误

| 错误 | 修复 |
|------|------|
| 表演性同意 | 陈述要求或直接行动 |
| 盲目实施 | 首先根据代码库验证 |
| 批处理不测试 | 一次一个，每个测试 |
| 假设审查者正确 | 检查是否破坏东西 |
| 避免反驳 | 技术正确性 > 舒适 |
| 部分实施 | 首先澄清所有项目 |
| 无法验证，无论如何继续 | 陈述限制，请求方向 |

### 2.11 真实示例

**表演性同意（不好）：**
```
审查者："删除遗留代码"
❌ "你完全正确！让我删除它..."
```

**技术验证（好）：**
```
审查者："删除遗留代码"
✅ "检查中... 构建目标是 10.15+，此 API 需要 13+。向后兼容需要遗留。当前实现有错误的 bundle ID - 修复它还是放弃 pre-13 支持？"
```

**YAGNI（好）：**
```
审查者："正确实施指标跟踪，包括数据库、日期过滤器、CSV 导出"
✅ "在代码库中 grep - 没有任何东西调用此端点。删除它（YAGNI）？还是有我遗漏的使用？"
```

## 三、审查模板

### 3.1 审查请求模板

```
我已完成 [功能名称] 的实现，请求代码审查。

## 实现内容
{WHAT_WAS_IMPLEMENTED}

## 需求文档
{PLAN_OR_REQUIREMENTS}

## 提交范围
- BASE_SHA: {BASE_SHA}
- HEAD_SHA: {HEAD_SHA}

## 简要描述
{DESCRIPTION}
```

### 3.2 审查反馈模板

```
## 代码审查报告

### 提交范围
- BASE_SHA: {BASE_SHA}
- HEAD_SHA: {HEAD_SHA}

### 优势
- [列出代码优点]

### 问题
| 优先级 | 描述 | 位置 |
|--------|------|------|
| 关键 | [必须修复] | [文件:行号] |
| 重要 | [应该修复] | [文件:行号] |
| 次要 | [建议改进] | [文件:行号] |

### 评估
- [ ] 准备好继续
- [ ] 需要修复后再审查
- [ ] 有重大问题需要讨论
```

---

## 底线

**外部反馈 = 要评估的建议，而不是要遵循的命令。**

验证。提问。然后实施。

没有表演性同意。技术严谨性始终存在。
