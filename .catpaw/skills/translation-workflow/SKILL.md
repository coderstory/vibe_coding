# 文档翻译工作流

本技能提供文档翻译的标准流程和规范，确保翻译质量和术语一致性。

---

## 一、翻译原则

### 1.1 核心原则

1. **准确传达原意**：翻译应忠实于原文，不随意增删内容
2. **符合中文表达习惯**：避免翻译腔，使用自然的中文表达
3. **术语一致性**：同一术语全文使用相同翻译
4. **保持格式完整**：保留原文的 Markdown 格式、链接、代码块

### 1.2 不翻译的内容

| 类型 | 说明 |
|------|------|
| 代码块 | 代码内容不翻译，仅翻译注释 |
| 命令 | 如 `npm install`、`git commit` |
| 配置项 | 如 `package.json` 中的字段名 |
| URL | 网址和路径不翻译 |
| 专有名词 | 如 `Vue`、`React`、`TypeScript` |
| 技术术语 | 约定保留英文的术语 |

## 三、翻译格式规范

### 3.1 Markdown 格式

保持原文格式不变：
- 标题层级（#, ##, ###）
- 列表（-, 1.）
- 表格（|---|）
- 代码块（```）
- 链接（[]()）
- 图片（![]()）

### 3.2 代码块处理

````markdown
<!-- 原文 -->
```javascript
// This is a comment
const name = "test";
```

<!-- 翻译后 -->
```javascript
// 这是一个注释
const name = "test";
```
````

### 3.3 链接处理

保持链接地址不变：

```markdown
<!-- 原文 -->
See [documentation](https://example.com/docs) for details.

<!-- 翻译后 -->
详情请参阅 [文档](https://example.com/docs)。
```

## 四、翻译工作流

### 4.1 准备阶段

```
1. 确定翻译范围
   - 列出需要翻译的文件
   - 按优先级排序

2. 创建翻译分支
   git checkout -b docs/translate-to-zh

3. 更新术语表
   - 检查是否有新术语需要统一
   - 记录特殊术语的处理方式
```

### 4.2 翻译阶段

```
1. 逐文件翻译
   - 先翻译小文件建立风格
   - 大文件分段翻译

2. 翻译要点
   - 标题和说明文字完整翻译
   - 代码注释翻译
   - 代码本身不翻译
   - 保持链接和引用有效

3. 标记进度
   - 记录已翻译的文件
   - 标记待审核的部分
```

### 4.3 验证阶段

```
1. 格式验证
   - Markdown 渲染正常
   - 表格对齐
   - 代码块高亮正确

2. 术语验证
   - 术语使用一致
   - 检查术语对照表

3. 链接验证
   - 所有链接可访问
   - 文件引用路径正确
```

### 4.4 提交阶段

```
1. 提交翻译
   git add .
   git commit -m "docs(i18n): 翻译 xxx 文档为中文"

2. 创建 PR
   gh pr create --title "docs: 翻译文档为中文" --body "翻译说明..."

3. 审核合并
   - 团队审核术语翻译
   - 修改审核意见
   - 合并到主分支
```

## 五、常见翻译问题

### 5.1 避免的翻译

| ❌ 错误 | ✅ 正确 |
|---------|---------|
| "你可以..." (过于口语) | "用户可以..." 或直接省略主语 |
| "它将会..." (翻译腔) | "将..." |
| "在...的情况下" (啰嗦) | "当...时" 或直接省略 |
| "进行...操作" (冗余) | 直接使用动词 |

### 5.2 语气规范

- 使用第二人称："你" 而非 "您"
- 使用简洁的书面语
- 避免过度口语化

### 5.3 特殊情况处理

**多义词**：根据上下文选择合适的翻译
```
"Right" 可以翻译为：
- 右边（方向）
- 正确（准确）
- 权利（法律）
```

**无标准翻译的术语**：保持英文，括号备注中文
```
Edge Case（边缘情况）
Boilerplate（样板代码）
```
