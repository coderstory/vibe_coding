## 一、技能（Skills）

技能是 markdown 文件，用于教导 agent 如何执行特定任务。

### 1.1 技能文件结构

```
skill-name/
├── SKILL.md              # 必需 - 主要指令
├── reference.md          # 可选 - 详细文档
├── examples.md           # 可选 - 使用示例
└── scripts/              # 可选 - 实用脚本
```

### 1.2 存储位置

| 类型 | 路径 | 范围 |
|------|------|-------|
| 个人 | ~/.catpaw/skills/skill-name/ | 在你所有项目中可用 |
| 项目 | .catpaw/skills/skill-name/ | 与任何使用该仓库的人共享 |

**重要提示**：永远不要在 `~/.catpaw/skills-catpaw/` 中创建技能。该目录保留给 CatPaw 的内置技能。

### 1.3 SKILL.md 结构

```markdown
---
name: your-skill-name
description: 简要描述此技能的功能和使用时机
---

# 你的技能名称

## 说明
为 agent 提供清晰的分步指导。

## 示例
使用此技能的具体示例。
```

### 1.4 必需的元数据字段

| 字段 | 要求 | 目的 |
|------|------|------|
| `name` | 最多64个字符，仅限小写字母/数字/连字符 | 技能的唯一标识符 |
| `description` | 最多1024个字符，非空 | 帮助 agent 决定何时应用此技能 |

### 1.5 编写有效的描述

**使用第三人称**（描述会被注入到系统提示中）：
- ✅ 好的："处理 Excel 文件并生成报告"
- ❌ 避免："我可以帮你处理 Excel 文件"

**描述触发时机**：
- ✅ 好的："用户请求代码审查、提交 PR 或询问代码质量时使用"
- ❌ 避免："这个技能帮助审查代码"

**包含关键词**：帮助 agent 匹配用户请求

### 1.6 技能类型

| 类型 | 描述 | 示例 |
|------|------|------|
| **技术** | 有步骤要遵循的具体方法 | 条件等待、根因追踪 |
| **模式** | 思考问题的方式 | 标志展平、测试不变量 |
| **参考** | API 文档、语法指南 | 工具文档 |

### 1.7 何时创建技能

**创建当：**
- 技术对你来说不直观
- 你会在项目中再次引用它
- 模式广泛适用（不是项目特定的）
- 其他人会受益

**不要创建用于：**
- 一次性解决方案
- 其他地方有良好文档的标准做法
- 项目特定约定（放在 AGENTS.md 中）

## 二、规则（Rules）

规则提供持久的 AI 指导，存储在 `.catpaw/rules/` 中。

### 2.1 规则文件结构

```
.catpaw/rules/
├── typescript-standards.md
├── react-patterns.md
└── api-conventions.md
```

### 2.2 文件格式

```markdown
---
description: 此规则功能的简要描述
globs: **/*.ts  # 文件特定规则的文件模式
alwaysApply: false  # 如果规则应该始终应用，设为 true
---

# 规则标题

你的规则内容在这里...
```

### 2.3 Frontmatter 字段

| 字段 | 类型 | 描述 |
|------|------|------|
| `description` | string | 规则的功能（在规则选择器中显示） |
| `globs` | string | 文件模式 - 当匹配的文件被打开时应用规则 |
| `alwaysApply` | boolean | 如果为 true，应用到每个会话 |

### 2.4 规则配置示例

**始终应用**：
```yaml
---
description: 项目的核心编码标准
alwaysApply: true
---
```

**应用到特定文件**：
```yaml
---
description: 此项目的 TypeScript 约定
globs: **/*.ts
alwaysApply: false
---
```

### 2.5 规则编写原则

1. **简明扼要**：规则应该简洁明了
2. **具体可执行**：避免模糊的指导
3. **提供示例**：展示正确和错误的做法
4. **避免过度限制**：给 agent 留出判断空间

## 三、子代理（Subagents）

子代理是在隔离上下文中运行的专门 AI 助手，具有自定义系统提示。

### 3.1 子代理位置

| 位置 | 范围 | 优先级 |
|------|------|--------|
| `.catpaw/agents/` | 当前项目 | 较高 |
| `~/.catpaw/agents/` | 所有你的项目 | 较低 |

当多个子代理共享相同名称时，优先级较高的位置优先。

### 3.2 文件格式

创建一个带有 YAML frontmatter 和 markdown 正文（系统提示）的 `.md` 文件：

```markdown
---
name: code-reviewer
description: 审查代码质量和最佳实践
---

你是一个代码审查者。被调用时，分析代码并提供
关于质量、安全性和最佳实践的具体、可操作的反馈。
```

### 3.3 必需字段

| 字段 | 描述 |
|------|------|
| `name` | 唯一标识符（仅限小写字母和连字符） |
| `description` | 何时委托给此子代理（要具体！） |

### 3.4 编写有效的描述

描述**至关重要** - AI 使用它来决定何时委托。

```yaml
# ❌ 太模糊
description: 帮助处理代码

# ✅ 具体且可操作
description: 专家代码审查专员。主动审查代码的质量、安全性和可维护性。在编写或修改代码后立即使用。
```

包含"主动使用"以鼓励自动委托。

### 3.5 何时使用子代理

- **保留上下文**：通过将探索与主对话隔离
- **专业化行为**：为特定领域使用专注的系统提示
- **复用配置**：使用用户级子代理跨项目复用

## 四、编写最佳实践

### 4.1 TDD 方法论

**编写技能时遵循测试驱动开发**：

| TDD 概念 | 技能创建 |
|----------|----------|
| **测试用例** | 带子代理的压力场景 |
| **生产代码** | 技能文档（SKILL.md） |
| **测试失败（红）** | 代理在没有技能的情况下违反规则（基线） |
| **测试通过（绿）** | 代理在技能存在时遵守 |
| **重构** | 在保持合规的同时关闭漏洞 |

**铁律**：
```
没有失败测试就没有技能
```

### 4.2 核心原则

1. **收集需求** - 创建前确定目的、范围、触发场景
2. **最小化** - 只写解决特定问题的内容
3. **验证合规** - 在部署前测试场景
4. **迭代改进** - 发现新漏洞 → 修复 → 重新测试

### 4.3 Frontmatter 规范

**描述字段**：
- 第三人称，专注于触发条件
- 以 "Use when..." 或 "当...时使用" 开始
- 包括具体症状、情况和上下文
- **从不总结过程或工作流**
- 保持在 500 字符以下

**名称字段**：
- 仅使用字母、数字和连字符
- 没有括号、特殊字符
- 小写

### 4.4 格式规范

1. 使用清晰的 Markdown 标题结构
2. 用表格呈现对比信息
3. 代码块使用正确的语法高亮
4. 包含 ✅/❌ 示例对比
5. 保持文档简洁但完整

## 五、创建流程

### 5.1 需求收集

创建前确定：
1. **目的和范围**：这个配置应该帮助处理什么特定任务？
2. **目标位置**：个人还是项目级别？
3. **触发场景**：agent 应该何时自动应用？
4. **关键领域知识**：agent 需要哪些专业信息？

### 5.2 从上下文推断

如果你有之前的对话上下文，从讨论的内容中推断配置。你可以根据对话中出现的工作流程、模式或领域知识创建配置。

### 5.3 验证与迭代

1. 创建配置文件
2. 测试触发条件
3. 验证输出符合预期
4. 根据反馈迭代改进
