## 简介

### 什么是 Superpowers？

Superpowers 是一个结构化的 AI 编程框架，通过**强制技能调用**和**严格工作流程**确保开发质量。

### 核心理念

- **纪律优于效率** — 遵循流程比走捷径更能保证质量
- **证据优于断言** — 验证后再声称完成
- **设计先于实现** — 理解需求后再写代码

## 核心原则

### 技能优先级

| 优先级 | 类型 | 说明 |
|--------|------|------|
| **1** | 用户明确指令 | CLAUDE.md、GEMINI.md、AGENTS.md 的要求最高 |
| **2** | Superpowers 技能 | 覆盖默认行为的技能规则 |
| **3** | 默认系统提示 | 最低优先级 |

### 技能触发规则

**任何任务都有 1% 可能触发技能。**

```
收到用户消息
    ↓
有技能可能适用？
    ↓ 是
加载技能 → 遵循执行
    ↓ 否
直接响应
```

**触发信号（停止并检查技能）：**

| 想法 | 现实 |
|------|------|
| "这只是个简单问题" | 问题即任务，需要检查技能 |
| "我需要先收集上下文" | 技能检查先于澄清问题 |
| "我可以直接查看文件" | 技能告诉你如何探索 |
| "我快速检查一下" | 技能告诉你如何验证 |
| "这不需要正式技能" | 存在技能就必须使用 |
| "我记住了这个技能" | 技能会更新，每次都要读取 |
| "这不算任务" | 行动=任务，需要检查 |
| "技能有点杀鸡用牛刀" | 简单的事容易变复杂 |
| "我先做这一件事" | 先检查技能 |
| "我感觉这很有效率" | 无纪律的行动浪费时间 |

## 技能列表

### 过程技能（确定如何工作）

#### brainstorming
**何时使用：** 任何创造性工作之前 — 创建功能、构建组件、添加功能或修改行为。

**工作流程：**
```
探索项目上下文 → 提出视觉伴侣（如需要）→ 逐一提问 → 提出2-3种方案 → 展示设计 → 编写设计文档 → 规格自审 → 用户审核 → 调用 writing-plans
```

**核心规则：**
- 绝不在用户批准设计前开始实现
- 每个问题单独发送
- 多选问题优先
- YAGNI（你不需要它）原则
- 提出2-3种方案并附带权衡

#### systematic-debugging
**何时使用：** 遇到任何 bug、测试失败或意外行为。

**铁律：** `没有根因调查不得修复`

**四阶段流程：**

| 阶段 | 活动 | 成功标准 |
|------|------|----------|
| **1. 根因调查** | 读错误信息、重现、检查变更、收集证据 | 理解问题和原因 |
| **2. 模式分析** | 找类似代码、对比参考、识别差异 | 找到差异 |
| **3. 假设和测试** | 形成理论、最小化测试 | 确认或新假设 |
| **4. 实现** | 创建测试、修复、验证 | Bug修复、测试通过 |

**关键规则：**
- 3次修复失败后停止，质疑架构
- 不要"快速修复"——系统性调试更快

#### test-driven-development
**何时使用：** 实现任何功能或 bug 修复，在写实现代码之前。

**铁律：** `没有失败的测试就不能写生产代码`

**红绿重构循环：**

```
RED    → 写失败测试
  ↓
验证失败
  ↓
GREEN  → 写最小代码使测试通过
  ↓
验证通过
  ↓
REFACTOR → 清理（保持测试绿色）
  ↓
下一个 RED
```

**核心原则：**
- 测试必须在实现前写
- 看到测试失败才能证明测试有效
- 测试通过后删除并用 TDD 重写

#### verification-before-completion
**何时使用：** 声称工作完成、修复成功或通过之前，在提交或创建 PR 之前。

**铁律：** `没有新鲜验证证据不得声称完成`

**流程：**
```
识别验证命令 → 执行完整命令 → 读取输出 → 验证是否确认声称
    ↓
是 → 声称（有证据）
    ↓
否 → 说明实际状态（带证据）
```

### 实现技能（指导执行）

#### writing-plans
**何时使用：** 有规格或需求的多步骤任务，在接触代码之前。

**输出：** `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`

**计划结构：**
- 每个任务 2-5 分钟
- 完整代码示例
- 精确文件路径
- 验证命令和预期输出

**核心规则：**
- 无占位符（TBD、TODO、"implement later"）
- 每个步骤显示实际代码
- 自审检查：规格覆盖、占位符扫描、类型一致性

#### subagent-driven-development
**何时使用：** 在当前会话中执行有独立任务的实现计划。

**流程：**
```
读取计划 → 提取所有任务 → 为每个任务创建 TodoWrite
    ↓
每个任务：
  1. 分派实施者子代理
  2. 实施者实现、测试、提交、自审
  3. 分派规格合规审查
  4. 规格审查通过？→ 修复问题
  5. 分派代码质量审查
  6. 质量审查通过？→ 修复问题
    ↓
所有任务完成 → 分派最终代码审查 → 使用 finishing-a-development-branch
```

**两阶段审查：** 规格合规 → 代码质量

#### executing-plans
**何时使用：** 在单独会话中执行书面实现计划（有检查点）。

**流程：**
```
步骤 1: 加载和审查计划
  - 读取计划文件
  - 识别问题或顾虑
  - 有顾虑？→ 向用户提出
  
步骤 2: 执行任务
  - 标记进行中
  - 严格遵循计划步骤
  - 运行验证
  - 标记完成

步骤 3: 完成开发
  - 使用 finishing-a-development-branch
```

#### finishing-a-development-branch
**何时使用：** 实现完成、所有测试通过，需要决定如何集成工作时。

**流程：**
```
步骤 1: 验证测试（必须先通过）
步骤 2: 确定基础分支
步骤 3: 呈现选项
  1. 本地合并到 <base-branch>
  2. 推送并创建 Pull Request
  3. 保持分支（稍后处理）
  4. 丢弃此工作
步骤 4: 执行选择
步骤 5: 清理 worktree
```

### 协作技能

#### receiving-code-review
**何时使用：** 收到代码审查反馈，在实现建议之前。

**流程：**
```
读取反馈 → 理解（用自己的话重述）→ 验证代码库现实 → 评估（对这个代码库技术上合理？）→ 回应 → 实现
```

**核心规则：**
- 绝不表演性同意（ "你说得对！"）
- 验证后再实现
- 技术正确性 > 社交舒适
- 不清晰时先问

#### requesting-code-review
**何时使用：** 完成任务、重大功能实现后，或合并前验证工作。

---

### 工作流技能

#### dispatching-parallel-agents
**何时使用：** 面对 2+ 个可并行处理的无共享状态或顺序依赖的独立任务。

#### using-git-worktrees
**何时使用：** 需要与当前工作隔离的工作区，或在执行实现计划之前。

## 核心工作流

### 完整功能开发流程

```
┌─────────────────────────────────────────────────────────────┐
│  brainstorming                                             │
│  理解需求 → 探索方案 → 用户批准设计                         │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  writing-plans                                             │
│  创建详细实现计划（带任务分解、代码示例、验证命令）          │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  subagent-driven-development 或 executing-plans              │
│  执行计划 → 逐任务验证 → 规格+质量双重审查                   │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  finishing-a-development-branch                             │
│  验证测试 → 呈现合并选项 → 执行选择 → 清理                  │
└─────────────────────────────────────────────────────────────┘
```

### Bug 修复流程

```
┌─────────────────────────────────────────────────────────────┐
│  systematic-debugging                                      │
│  根因调查 → 模式分析 → 假设测试 → 实现修复                  │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  test-driven-development (创建回归测试)                     │
│  写失败测试 → 写修复 → 验证通过                             │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  verification-before-completion                              │
│  运行验证 → 确认修复 → 报告实际状态                           │
└─────────────────────────────────────────────────────────────┘
```

## 常见场景

### 场景 1: 新功能开发

```
你：我们要添加用户认证功能

AI：[调用 brainstorming 技能]
    - 探索项目上下文
    - 逐一提问澄清需求
    - 提出 2-3 种方案
    - 展示设计并获得批准
    - 编写设计文档到 docs/superpowers/specs/...

[用户批准设计]

AI：[调用 writing-plans 技能]
    - 分析设计文档
    - 创建任务分解
    - 每个任务含代码示例和验证命令
    - 保存到 docs/superpowers/plans/...

[提供执行选项]

用户：使用 subagent-driven

AI：[调用 subagent-driven-development]
    - 为每个任务分派子代理
    - 规格+质量两阶段审查
    - 完成后调用 finishing-a-development-branch
```

### 场景 2: Bug 修复

```
你：登录按钮不工作

AI：[调用 systematic-debugging 技能]
    - 读错误信息
    - 尝试重现
    - 检查最近变更
    - 追踪数据流找到根因
    
[找到根因：token 过期处理缺失]

AI：[调用 test-driven-development]
    - 写失败测试重现 bug
    - 写最小修复代码
    - 验证测试通过
    
[调用 verification-before-completion]
    - 运行完整测试套件
    - 确认修复有效
    - 报告实际状态
```

### 场景 3: 代码审查

```
审查者：建议添加错误处理和使用装饰器模式

AI：[调用 receiving-code-review 技能]
    - 验证建议是否适合此代码库
    - 检查当前实现和使用情况
    - 技术评估：
      * 装饰器模式在这里是否过度设计？
      * YAGNI 检查：grep 代码库，有调用这个函数吗？
    - 推送有技术理由的反建议，或开始实现
```
