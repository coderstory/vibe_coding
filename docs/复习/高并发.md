### 秒杀系统
1. 独立服务器
>独立微服务，数据库，redis，mq
>
> 不影响其他服务，方便扩容和收缩
2. 防止恶意攻击 （内部人员自己去秒杀 防止暴露链路）
> 前端不直接显示秒杀按钮 / 接口地址，用户校验通过后动态生成加密接口
>
> 后端生成一次性签名（sign），包含用户 ID、商品 ID、时间戳、随机串，只有签名校验通过才允许访问秒杀接口
>
> 秒杀地址一次性有效，防止爬虫复用接口
>
> 日志审计：记录所有秒杀请求，内部人员秒杀行为留痕、风控拦截
>
> 禁用秒杀接口直接抓包访问
>
> 网关层屏蔽敏感请求头，不返回异常堆栈信息

3. 库存处理 （预热 快速减扣）（不使用数据库检验库存，使用redis的信号量控制秒杀数量 ）
>秒杀开始前，通过定时任务将商品库存从数据库加载到 Redis
>
>Redis 存储结构：seckill:stock:{商品ID} = 库存数量
>
> 使用 Redis Lua 脚本 实现原子性扣减（单命令执行，杜绝超卖）
>
>核心逻辑：判断库存 > 0 → 扣减 1 → 返回成功，全程 Redis 内存操作，QPS 可达 10w+
禁止直接用数据库查询 / 更新库存（数据库抗不住高并发）
>
> Redis 扣减成功后，发送消息到 MQ，异步更新数据库库存
>
> 最终一致性：定时任务校对 Redis 和数据库库存，补齐差值
4. 动静分离 CDN分担压力
> 静态资源全部从CDN获取
>
> 秒杀商品详情页做页面级缓存（Redis 缓存页面片段），减少服务渲染压力
>
5. 恶意请求拦截 识别非法请求
>IP 限流：同一 IP 1 秒内请求超过阈值直接拉黑
>
>设备指纹拦截：同一设备频繁请求、模拟器请求直接拒绝
>
>请求参数校验：过滤空参数、非法商品 ID、伪造用户 ID
>
>黑名单机制：Redis 存储黑名单（IP / 用户 ID），网关层优先校验
>
>羊毛党识别：同一账号短时间多次秒杀、批量账号秒杀，直接拦截
>
6. 流量错峰 校验码，购物车 分摊流量到更长的时间线
>验证码 / 滑块验证
>
>秒杀前必须完成图形验证码 / 滑块验证，人为拉长请求时间，打散瞬间流量
>
>购物车预热
>
>用户提前加入购物车，秒杀时从购物车一键下单，分摊预热期流量
>
>分时段秒杀
>
>同一商品分多场秒杀，避免流量集中在同一秒
>
>排队机制
>
>高并发时返回排队中，前端轮询结果，削平峰值
7. 限流，熔断，降级
>限流
>
>网关层：限制秒杀接口总 QPS（例如 1 万 / 秒）
>
>服务层：限制单用户、单商品的请求频率
>
>熔断
>
>秒杀服务出现异常、超时，自动熔断，停止接收新请求，快速失败
>
>降级
>
>流量超阈值时，降级非核心功能（不查询商品详情、不记录日志）
>
>直接返回 “秒杀结束”，保护服务不崩溃
>
>隔离
>
>Sentinel 线程池隔离，秒杀请求不占用其他服务线程
8. 引入消息中间件 （削峰填谷）
>削峰
>
>秒杀请求不直接写入数据库，先写入 MQ 队列，消费者匀速处理，防止数据库被打垮
>
>解耦
>
>秒杀服务 → MQ → 订单服务 → 库存服务，微服务之间不直接同步调用
>
>异步处理
>
>Redis 扣减库存成功 → 发送消息到 MQ → 异步创建订单、更新库存
>
>前端只需知道 “排队中”，无需等待数据库执行完成
9. 限时订单
>超时未支付自动取消
>
>订单创建后，发送延迟消息到 MQ（例如 15 分钟）
>
>超时回补库存
>
>消费者收到延迟消息，检查订单状态：
>
>未支付 → 取消订单 → Redis 库存 +1 → 数据库库存同步
>
>Redis 过期监听
>
>辅助方案：用 Redis Key 过期事件监听订单超时，双重保障
>
> 使用MQ的延迟消息
>
> 具体订单到了限时时间后检查是否支付
>
10. 维护秒杀商品的上架 定时任务设置库存和开启秒杀
> 定时上架
>
> 活动开始前 N 分钟，自动将商品状态改为 “可秒杀”
>
> 库存预热
>
> 从数据库同步库存到 Redis
>
> 自动开启 / 关闭秒杀
>
> 定时修改秒杀状态，防止提前秒杀、超时未关闭
>
> 活动后数据清理
>
> 自动归档订单、清理 Redis 临时缓存
>
11. 秒杀业务的主流程一般不直接访问数据库，而是通过redis和MQ快速完成
>全程不直接访问数据库，Redis + MQ 支撑核心流程，mq需要限制队列长度，也就是限制最大的处理订单量，超出的直接返回系统繁忙
```
>前端请求 → Spring Cloud Gateway（拦截、限流、鉴权） →
>seckill-service（Redis Lua 原子扣减库存） →
>成功 → 发送MQ消息 → 异步创建订单/更新库存
>失败 → 直接返回“库存不足/秒杀失败”
```
>流程特点
>
>所有判断、扣减都在 Redis 完成，微服务只做逻辑转发
>
>数据库只做最终数据落地，不参与高并发核心流程
>
>微服务异步化、无状态化，支持高并发
>
12. 二级秒杀系统（淘宝）
> 一级是商品抢购页面限制，无库存不能加购
>
> 二级是购物车入付款的时候，无库存不能下单

13.排队系统（小米）
> 用户点击抢购，会弹出正在排队中的弹窗，此时用户无法再次点击抢购按钮。
>
> 此时用户的请求进入队列
>
> 这个请求可能随时被后端会被丢弃，也可能真的正在执行下单流程
>
> 如果超时或者被丢弃，则在N秒后返回失败
>
> 需要浏览器记录秒杀id，用户刷新页面再次秒杀，只会重新轮询之前的秒杀id
>
> 使用redis限制用户只能一次秒杀，key：seckill:req:{userId}:{goodsId}:{sessionId}
> 
14. 库存扣减流程
```graph TD
    A[用户下单请求] --> B(API网关: 限流/鉴权);
    B --> C(订单服务: 创建订单 Pending);
    C --> D[Redis: 库存预扣减 Lua脚本];
    D -- 成功 --> E[RocketMQ: 发送事务消息 Half Message];
    E --> F{订单服务: 本地事务提交?};
    F -- 是 --> G[RocketMQ: Commit];
    F -- 否 --> H[RocketMQ: Rollback];
    G --> I[库存服务: 消费消息];
    I --> J[MySQL: 库存扣减 乐观锁];
    J --> K[库存服务: 向订单服务发送确认];
    K --> L[订单服务: 更新订单状态为 Confirmed];
    L --> M[通知服务: 异步发送通知]; 
```

# 缓存穿透、缓存击穿、缓存雪崩 详解
## 一、缓存穿透
### 概念
大量请求查询**数据库中不存在的数据**，请求无法命中缓存，全部直接访问数据库，引发数据库压力剧增。

### 解决方案
1. 缓存空值：为不存在的数据缓存空值，配置短过期时间，拦截重复无效请求
2. 布隆过滤器：在缓存层前置布隆过滤器，从源头过滤不存在的Key

---

## 二、缓存击穿
### 概念
**单个热点Key**缓存过期的瞬间，海量并发请求直接击穿缓存，全部请求数据库。

### 解决方案
1. 热点Key永不过期：核心热点数据不设置过期时间，异步更新缓存
2. 分布式互斥锁：仅允许一个线程查询数据库并重建缓存，其他线程等待

---

## 三、缓存雪崩
### 概念
**大量Key同一时间集中过期** 或 缓存服务宕机，所有请求直击数据库，导致数据库崩溃。

### 解决方案
1. 随机过期时间：给缓存TTL添加随机偏移量，避免批量同时过期
2. 服务高可用：Redis集群、主从、哨兵模式，防止缓存服务单点故障
3. 多级缓存：本地缓存(Caffeine) + 分布式缓存(Redis) 双重兜底
4. 限流熔断：网关层限流、服务熔断，保护数据库

---

## 四、核心区别对比
| 问题类型 | 核心诱因 | 影响范围 |
| ---- | ---- | ---- |
| 缓存穿透 | 查询不存在的数据 | 全量无效请求，长期攻击 |
| 缓存击穿 | 单个热点Key过期 | 单点热点，瞬时流量 |
| 缓存雪崩 | 大量Key集中过期/服务宕机 | 全局大规模，数据库雪崩 |

# 缓存选型对比：Redis vs MongoDB
| 维度 | Redis | MongoDB |
| ---- | ---- | ---- |
| **定位** | 高性能分布式内存数据库，支持多种数据结构（String、Hash、List、Set、Zset等），提供持久化、主从复制，集群等功能 | 面向文档的分布式数据库，以BSON格式存储数据，支持索引、聚合、分片、副本集等。适合结构化或半结构化数据存储 |
| **适用场景** | 1. 高速缓存层，加速热点数据的访问<br>2. 分布式锁：利用原子操作实现分布式锁<br>3. 实时数据：会话存储，排名榜、实时消息队列<br>4. 轻量级存储：数据比较小，读写多写少 | 1. 结构复杂或关联查询的缓存数据<br>2. 大数据缓存：大数据缓存避免内存过大<br>3. 历史数据缓存<br>4. 缓存与业务数据一致 |
| **特点** | 1. 内存存储：读写性能高（微秒）<br>2. 数据结构丰富：支持多种数据模型，适用场景灵活<br>3. 分布式支持：集群、分片、高可用、高并发<br>4. 持久化可选：RDB、AOF、混合持久化 | 1. 磁盘存储为主：数据量大，成本相对低<br>2. 查询功能强大：支持复杂查询、聚合、全文检索<br>3. 水平拓展灵活：分片支持海量数据存储<br>4. 数据模型灵活：字段动态拓展、机构易变 |
