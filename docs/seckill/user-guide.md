# 秒杀系统用户使用手册

## 1. 系统概述

秒杀系统是一个高并发电商活动平台，支持用户抢购限时特价商品。系统采用多层保护机制，确保在海量并发请求下仍能稳定运行。

### 1.1 主要功能模块

| 模块 | 功能说明 |
|------|----------|
| 秒杀首页 | 展示所有进行中或即将开始的秒杀活动 |
| 活动详情 | 查看活动详细信息，包括开始时间、结束时间、限购数量等 |
| 抢购记录 | 查看历史抢购订单和状态 |
| 购物车 | 管理秒杀商品购物车 |
| 我的订单 | 查看和管理订单，支持支付和取消 |
| 监控大盘 | 实时监控系统运行状态（管理员功能） |

## 2. 快速入门

### 2.1 访问系统

1. 打开浏览器访问系统地址
2. 输入用户名和密码登录
3. 登录成功后进入主界面

### 2.2 参与秒杀活动

**步骤 1：浏览秒杀活动**
- 在导航菜单中选择"秒杀首页"
- 查看当前可用的秒杀活动列表

**步骤 2：查看活动详情**
- 点击活动卡片进入详情页面
- 查看活动规则、时间、价格等信息

**步骤 3：参与抢购**
- 确认活动状态为"进行中"
- 点击"立即抢购"按钮
- 系统会自动进行签名验证和库存检查

**步骤 4：等待结果**
- 抢购成功后，系统会返回排队编号
- 可通过排队编号查询抢购结果

## 3. 功能详解

### 3.1 秒杀首页

秒杀首页展示所有秒杀活动，包含以下信息：
- 活动名称
- 活动状态（未开始/进行中/已结束）
- 开始时间
- 结束时间
- 限购数量

**状态标识说明：**
| 状态 | 标识 | 说明 |
|------|------|------|
| 未开始 | 灰色标签 | 活动尚未开始，无法抢购 |
| 进行中 | 绿色标签 | 活动正在进行，可抢购 |
| 已结束 | 灰色标签 | 活动已结束 |

### 3.2 活动详情

活动详情页提供完整的活动信息：
- 活动名称和描述
- 开始/结束时间
- 每人限购数量
- 当前活动状态
- 抢购按钮（仅活动进行中可用）

### 3.3 抢购流程

抢购流程采用异步处理，步骤如下：

```
用户点击抢购
    ↓
系统验证签名（防篡改）
    ↓
检查用户是否在黑名单
    ↓
检查请求幂等性（防重复）
    ↓
Redis原子操作判断库存
    ↓
发送RocketMQ事务消息
    ↓
返回排队编号给用户
    ↓
异步创建订单并扣减库存
```

**状态说明：**
| 状态码 | 说明 |
|--------|------|
| 0 | 排队中，等待处理 |
| 1 | 抢购成功 |
| 2 | 抢购失败（库存不足/风控拦截等） |

### 3.4 抢购记录

抢购记录页面展示用户所有的抢购订单，包括：
- 订单号
- 商品信息
- 订单状态
- 下单时间

**订单状态说明：**
| 状态 | 说明 |
|------|------|
| 待支付 | 订单已创建，等待用户支付 |
| 已支付 | 支付完成，订单生效 |
| 已取消 | 用户主动取消订单 |
| 超时取消 | 支付超时自动取消 |

### 3.5 购物车

购物车功能允许用户：
- 查看已添加的秒杀商品
- 调整商品数量
- 移除商品
- 清空购物车

### 3.6 订单管理

在"我的订单"页面可以：
- 查看所有订单
- 对未支付订单进行支付
- 对未支付订单进行取消

**支付流程：**
1. 点击"支付"按钮
2. 确认支付信息
3. 完成支付

**取消流程：**
1. 点击"取消"按钮
2. 确认取消操作
3. 订单状态更新为"已取消"

### 3.7 监控大盘（管理员）

监控大盘实时展示系统运行状态：
- 当前并发处理数
- QPS统计
- 活动时间线

## 4. 常见问题

### Q1: 为什么抢购失败？

可能原因：
- 库存已售罄
- 活动尚未开始或已结束
- 您的IP被风控拦截
- 请求频率过高被限流
- 同一用户重复提交

### Q2: 如何查询抢购结果？

在抢购记录页面，输入排队编号或查看订单列表。

### Q3: 订单支付有时间限制吗？

是的，订单需要在规定时间内完成支付，否则会自动取消。

### Q4: 可以取消订单吗？

只有在"待支付"状态的订单可以取消，已支付的订单无法取消。

### Q5: 抢购时提示"系统繁忙"怎么办？

这是正常的限流保护机制，请稍后重试。

## 5. 安全说明

### 5.1 签名验证

每次抢购请求都需要携带签名，签名由后端生成，用于防止请求被篡改。

### 5.2 风控保护

系统内置多层风控：
- IP限流：同一IP每秒请求数限制
- 用户限流：同一用户并发请求限制
- 黑名单：违规用户将被加入黑名单

### 5.3 幂等性保证

系统保证同一幂等键的请求只会被处理一次，防止重复抢购。

## 6. 技术架构

系统采用三层保护机制：

| 层次 | 技术 | 作用 |
|------|------|------|
| 第一层 | Redis原子操作 | 快速过滤无效请求，高性能库存预扣减 |
| 第二层 | RocketMQ消息队列 | 削峰填谷，控制数据库写入并发 |
| 第三层 | 数据库乐观锁 | 最终一致性保障，防止超卖 |
