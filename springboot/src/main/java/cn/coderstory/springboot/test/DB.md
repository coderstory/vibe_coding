### 秒杀系统
1. 独立服务器
>独立微服务，数据库，redis，mq
>
> 不影响其他服务，方便扩容和收缩
2. 防止恶意攻击 （内部人员自己去秒杀 防止暴露链路）
> 前端不直接显示秒杀按钮 / 接口地址，用户校验通过后动态生成加密接口
>
> 后端生成一次性签名（sign），包含用户 ID、商品 ID、时间戳、随机串，只有签名校验通过才允许访问秒杀接口
> 
> 秒杀地址一次性有效，防止爬虫复用接口
> 
> 日志审计：记录所有秒杀请求，内部人员秒杀行为留痕、风控拦截
> 
> 禁用秒杀接口直接抓包访问
> 
> 网关层屏蔽敏感请求头，不返回异常堆栈信息

3. 库存处理 （预热 快速减扣）（不使用数据库检验库存，使用redis的信号量控制秒杀数量 ）
>秒杀开始前，通过定时任务将商品库存从数据库加载到 Redis
> 
>Redis 存储结构：seckill:stock:{商品ID} = 库存数量
> 
> 使用 Redis Lua 脚本 实现原子性扣减（单命令执行，杜绝超卖）
> 
>核心逻辑：判断库存 > 0 → 扣减 1 → 返回成功，全程 Redis 内存操作，QPS 可达 10w+
禁止直接用数据库查询 / 更新库存（数据库抗不住高并发）
> 
> Redis 扣减成功后，发送消息到 MQ，异步更新数据库库存
> 
> 最终一致性：定时任务校对 Redis 和数据库库存，补齐差值

> 静态资源全部从CDN获取

> 秒杀商品详情页做页面级缓存（Redis 缓存页面片段），减少服务渲染压力

>

# 缓存穿透、缓存击穿、缓存雪崩 详解
## 一、缓存穿透
### 概念
大量请求查询**数据库中不存在的数据**，请求无法命中缓存，全部直接访问数据库，引发数据库压力剧增。

### 解决方案
1. 缓存空值：为不存在的数据缓存空值，配置短过期时间，拦截重复无效请求
2. 布隆过滤器：在缓存层前置布隆过滤器，从源头过滤不存在的Key

---

## 二、缓存击穿
### 概念
**单个热点Key**缓存过期的瞬间，海量并发请求直接击穿缓存，全部请求数据库。

### 解决方案
1. 热点Key永不过期：核心热点数据不设置过期时间，异步更新缓存
2. 分布式互斥锁：仅允许一个线程查询数据库并重建缓存，其他线程等待

---

## 三、缓存雪崩
### 概念
**大量Key同一时间集中过期** 或 缓存服务宕机，所有请求直击数据库，导致数据库崩溃。

### 解决方案
1. 随机过期时间：给缓存TTL添加随机偏移量，避免批量同时过期
2. 服务高可用：Redis集群、主从、哨兵模式，防止缓存服务单点故障
3. 多级缓存：本地缓存(Caffeine) + 分布式缓存(Redis) 双重兜底
4. 限流熔断：网关层限流、服务熔断，保护数据库

---

## 四、核心区别对比
| 问题类型 | 核心诱因 | 影响范围 |
| ---- | ---- | ---- |
| 缓存穿透 | 查询不存在的数据 | 全量无效请求，长期攻击 |
| 缓存击穿 | 单个热点Key过期 | 单点热点，瞬时流量 |
| 缓存雪崩 | 大量Key集中过期/服务宕机 | 全局大规模，数据库雪崩 |

# 缓存选型对比：Redis vs MongoDB
| 维度 | Redis | MongoDB |
| ---- | ---- | ---- |
| **定位** | 高性能分布式内存数据库，支持多种数据结构（String、Hash、List、Set、Zset等），提供持久化、主从复制，集群等功能 | 面向文档的分布式数据库，以BSON格式存储数据，支持索引、聚合、分片、副本集等。适合结构化或半结构化数据存储 |
| **适用场景** | 1. 高速缓存层，加速热点数据的访问<br>2. 分布式锁：利用原子操作实现分布式锁<br>3. 实时数据：会话存储，排名榜、实时消息队列<br>4. 轻量级存储：数据比较小，读写多写少 | 1. 结构复杂或关联查询的缓存数据<br>2. 大数据缓存：大数据缓存避免内存过大<br>3. 历史数据缓存<br>4. 缓存与业务数据一致 |
| **特点** | 1. 内存存储：读写性能高（微秒）<br>2. 数据结构丰富：支持多种数据模型，适用场景灵活<br>3. 分布式支持：集群、分片、高可用、高并发<br>4. 持久化可选：RDB、AOF、混合持久化 | 1. 磁盘存储为主：数据量大，成本相对低<br>2. 查询功能强大：支持复杂查询、聚合、全文检索<br>3. 水平拓展灵活：分片支持海量数据存储<br>4. 数据模型灵活：字段动态拓展、机构易变 |
